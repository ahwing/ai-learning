对齐：RLHF / DPO

从能力到安全 — 让模型输出符合人类偏好

🌍 现实案例：ChatGPT 的"礼貌"和"安全"就来自 RLHF。OpenAI 雇佣了约 40 名标注员，通过对比排序让模型学会了什么是"好"的回答。这是从 GPT-3.5 到 ChatGPT 的核心技术跃迁。
🌍 现实案例：Meta 的 LLaMA 2 Chat 用 RLHF 训练了超过 5 轮迭代。每轮收集新的人类偏好数据，不断精调。而 LLaMA 3 则改用了 DPO，训练更稳定且无需单独的 RM。

DPO (Direct Preference Optimization)
1. 收集偏好对 (chosen/rejected)
↓
2. 直接优化策略（无需 RM）
↓
3. 隐式 reward = log ratio
✅ 无需单独的 Reward Model
✅ 训练稳定，像 SFT 一样简单
✅ LLaMA-3, Qwen-2 等使用
⚠️ 离线数据，无法在线探索
📐 DPO 核心公式
LDPO = -log σ(β · [log π(yw|x)/πref(yw|x) - log π(yl|x)/πref(yl|x)])
yw = 偏好回答 (chosen)  |  yl = 拒绝回答 (rejected)  |  β = 温度参数
本质：增大 chosen 的概率，减小 rejected 的概率，同时不偏离 reference model 太远

IPO
避免 DPO 的过拟合问题
使用平方 loss 替代 log-sigmoid
KTO
不需要成对数据
只需标记"好"或"坏"
ORPO
SFT + DPO 合并为一步
无需 reference model
SimPO
用平均 log prob 作为 reward
更简单，效果更好

🧠 核心概念深入

费曼式解释 — 用类比理解本质

🎯 "对齐"到底是什么？
类比：培训一个能力超强的新员工

想象你招了一个天才员工（预训练模型）。他读过所有书、知道所有知识，但他不知道公司的价值观：什么该做、什么不该做、怎样的回答才是"好"的。

SFT = 给他看工作范例："这类问题这样回答"
对齐 = 教他判断力："在这两个回答中，A 比 B 好，因为..."

为什么 SFT 不够？
• SFT 只教"做什么"，不教"为什么这样做更好"
• 模型可能学到表面模式而非深层偏好
• 面对新场景时，没有内化的"判断标准"来指导行为
对齐的本质：让模型内化一套评判标准，而非记忆固定答案
📊 Bradley-Terry 模型：把"谁更好"变成数学
类比：棋手等级分（Elo Rating）

你看两个棋手下棋，A 赢了 B。你能说 A 比 B 强，但"强多少"？Elo 系统给每个棋手一个分数，分差越大，强者赢的概率越高。

Bradley-Terry 模型做的事完全一样：给每个回答一个"分数"（reward），然后：
P(A 优于 B) = σ(r(A) - r(B)) = 1 / (1 + e-(r(A)-r(B)))
σ = sigmoid 函数，把分差映射到 [0,1] 概率
✓ 分差大 → 高概率认为 A 更好（确定性高）
✓ 分差小 → 接近 50/50（不确定）
✓ 只需要"谁比谁好"的相对判断，不需要绝对分数
这就是为什么偏好数据只需要标注"A 比 B 好"，而不需要打具体分数
🐕 KL 散度约束：为什么需要"牵绳"？
类比：遛狗的牵绳

把 SFT 模型想象成一条训练有素的狗（reference model），对齐训练就是让它学新技能。但你需要一根牵绳（KL 约束）：
🔗 绳太紧（β 太大）
模型几乎不能改变
学不到新的偏好
= 白训练了
✅ 绳刚好（β 适中）
模型能学习偏好
同时保持基础能力
= 理想状态
💨 绳太松（β 太小）
模型跑偏了
reward hacking
= 输出退化
实践经验：β 通常在 0.1-0.5 之间。太小模型会找到 reward 的漏洞（说一堆废话但 reward 高），太大则模型不会改变行为。
🎭 Reward Hacking：模型学会"作弊"
类比：应试教育 vs 真正学习

学生发现"写长一点分数就高"，于是疯狂凑字数而不关心内容质量。模型也会发现 reward model 的漏洞：
🔄 重复讨好
不断重复"希望这对您有帮助！"
RM 给高分但实际无用
📏 长度偏好
生成超长回答
因为 RM 倾向于给长回答高分
🎪 格式游戏
过度使用 markdown 格式
看起来"专业"但内容空洞
🦜 鹦鹉学舌
重复用户的问题再回答
制造"理解深刻"的假象
防御手段：KL 约束 + 更好的 RM + 多维度评估 + 定期更新 RM + 人工抽检

🌐 在线偏好优化 (Online)
用当前模型生成回答，实时收集偏好反馈
✓ 数据分布匹配当前策略
✓ 能持续改进，避免分布偏移
✗ 需要持续的人工/AI 标注
✗ 计算成本高（需要不断生成）
代表：Online RLHF, Online DPO, OAIF
📦 离线偏好优化 (Offline)
用预先收集的固定数据集训练
✓ 简单，一次收集多次使用
✓ 训练成本低，可复现
✗ 数据可能过时（分布偏移）
✗ 无法探索新的回答空间
代表：标准 DPO, IPO, KTO
📜 Constitutional AI：让模型自我批评
核心思想：不依赖人类标注，而是给模型一套"宪法"（原则），让它自己判断回答好坏。
1. 模型生成回答
↓
2. 模型根据"宪法"自我批评
↓
3. 模型修改回答（自我改进）
↓
4. 用改进前后的对比做 DPO/RLHF
优势：大幅减少人工标注需求 | 风险：模型可能有系统性盲点，自我批评不够客观
🔄 迭代 DPO 与自我博弈方法
核心思想：模型和自己的旧版本"对弈"，不断进化（类似 AlphaGo 的自我博弈）。
Iterative DPO
每轮用当前模型生成新回答
重新收集偏好 → 再训练
多轮迭代持续提升
SPIN (Self-Play)
当前模型 vs 上一轮模型
目标：区分人类写的和模型生成的
收敛时模型 ≈ 人类水平
Self-Rewarding
模型同时当 generator 和 judge
自己给自己打分
LLM-as-a-Judge 的极致

❓ 常见问题

对齐实践中最常被问到的问题

Q1: 需要多少偏好数据？
短答案：取决于任务复杂度，但通常比你想的少。

1K-5K
简单任务/微调
10K-50K
通用对齐
100K+
ChatGPT 级别
关键洞察：数据质量 > 数据数量。1000 条高质量、多样化的偏好对，胜过 10000 条噪声数据。优先确保标注一致性和场景覆盖度。
Q2: DPO 能完全替代 RLHF 吗？
短答案：大多数场景可以，但不是所有。

DPO 更好的场景：资源有限、需要快速迭代、数据量中等、追求训练稳定性
RLHF 更好的场景：需要在线探索新回答空间、任务非常复杂且多样、有充足计算资源

趋势：业界正在融合两者优势 — Online DPO（用 DPO 的简单性 + 在线数据生成）正成为主流选择。LLaMA-3、Qwen-2 等都采用了 DPO 或其变体。
Q3: 如何处理标注者分歧？
短答案：分歧是信号，不是噪声。

处理策略：
1. 多人标注 + 多数投票（至少 3 人，取 2/3 一致的）
2. 分歧大的样本单独分析 — 可能是主观偏好题，需要分场景处理
3. 使用 soft label（如 60% 选 A，40% 选 B）而非硬标签
4. 建立标注规范（annotation guideline），减少主观性

警告：强行统一有分歧的标注会引入偏见。承认不确定性比假装确定更好。
Q4: 什么是"对齐税"？如何最小化？
对齐税 (Alignment Tax)：为了安全性而牺牲的能力。

类比：给赛车装安全带和限速器 — 更安全了，但最高速度降低了。

表现形式：
• 过度拒绝（明明可以回答的问题也拒绝）
• 回答变得冗长啰嗦（加太多免责声明）
• 创造力下降（过于保守）

最小化策略：
1. 精细化安全边界 — 区分"真正有害"和"敏感但可回答"
2. 多目标优化 — 同时优化安全性和有用性
3. 分层对齐 — 先对齐有用性，再微调安全性
4. 持续评估 — 监控能力指标，发现退化及时修正
Q5: 如何检测和防止 Reward Hacking？
检测信号：
• Reward 分数持续上升，但人工评估质量下降
• 输出长度异常增加或格式过度统一
• 模型开始产生重复性的"讨好"模式

防御体系：
KL 约束
限制模型偏离程度
多维 Reward
分别评估有用/安全/格式
定期更新 RM
堵住已知漏洞
人工抽检
最终防线，发现新模式
核心原则：永远不要只看 reward 分数。建立独立于训练 reward 的评估体系（如人工评估、独立 RM、自动化测试集）。
