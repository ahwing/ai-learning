~30 min read Beginner friendly Updated June 2026

What is OpenAI Codex?

OpenAI Codex is an AI-powered coding assistant that helps developers write, debug, refactor, and understand code across multiple programming languages. It combines the power of GPT models with purpose-built tools for software development, offering a seamless experience across desktop apps, CLI tools, and IDE extensions.

Unlike simple autocomplete tools, Codex operates as an agentic coding assistant — it can understand project context, execute multi-step tasks, run shell commands, and deliver working code changes with minimal input from you.

Agentic Intelligence

Goes beyond autocomplete — plans, executes, and validates multi-step coding tasks autonomously.

Full Project Context

Understands your entire codebase, dependencies, and architectural patterns to deliver contextual suggestions.

Multi-Interface

Available as a Desktop App, CLI tool, and VS Code extension — use whichever fits your workflow.

Secure & Private

Code stays local. Models run in the cloud but your source code is not stored or used for training.

Who is this guide for?

This guide is designed for developers who are new to AI coding assistants, or those who want to set up Codex with Amazon Bedrock. No prior experience with AI tools is required.

Core Features

Codex ships with several powerful features that set it apart from simple code completion tools. Here's a deep dive into each one.

Goal Mode

Goal Mode is Codex's most powerful feature. Instead of chatting back and forth, you describe a high-level objective and Codex autonomously plans and executes the work — writing code, creating files, installing packages, and running tests.

How Goal Mode Works

1

You Describe the Goal

"Add user authentication with JWT tokens and a login page"

2

Codex Plans

Analyzes your codebase, creates a step-by-step execution plan, identifies files to create/modify.

3

Codex Executes

Writes code, installs dependencies, creates files, runs tests — all autonomously.

4

You Review

Review the changes as a unified diff, approve, request modifications, or reject.

Goal Mode Examples
# Simple goals codex "Fix the failing test in auth.test.js" codex "Refactor the database module to use connection pooling" # Complex goals codex "Add a REST API with CRUD endpoints for a 'products' resource, including validation, error handling, and unit tests" # Goals with constraints codex "Migrate from Express to Fastify without changing the public API interface"

Chat Mode

Chat Mode provides a conversational interface for when you want to discuss code, ask questions, or work iteratively on a problem. Think of it as pair programming with an AI teammate.

Ask Questions

"How does the auth middleware work?" — Get explanations of your own code or any concept.

Debug Together

Paste errors and stack traces, discuss root causes, and iterate on fixes collaboratively.

Skills

Skills are reusable instructions you can teach Codex. They act as templates or standard operating procedures that Codex follows when performing specific types of tasks.

~/.codex/skills/create-component.md
# Skill: Create React Component When asked to create a React component: 1. Create the component file in src/components/ 2. Use TypeScript with a Props interface 3. Include JSDoc comments on the exported component 4. Create a corresponding .test.tsx file with basic tests 5. Export from the components/index.ts barrel file 6. Follow naming convention: PascalCase for filenames 7. Use styled-components for styling (not CSS modules)

MCP (Model Context Protocol)

MCP allows Codex to connect to external tools and data sources — databases, APIs, documentation services, file systems, and more — extending its capabilities beyond just reading your local project files.

~/.codex/config.toml — MCP Server Configuration
# Connect Codex to a PostgreSQL database [[mcp_servers]] name = "postgres" command = "npx" args = ["@modelcontextprotocol/server-postgres", "postgresql://localhost/mydb"] # Connect to GitHub for PR/issue context [[mcp_servers]] name = "github" command = "npx" args = ["@modelcontextprotocol/server-github"] env = { GITHUB_TOKEN = "ghp_xxxxxxxxxxxx" } # Connect to a documentation search tool [[mcp_servers]] name = "docs" command = "npx" args = ["@modelcontextprotocol/server-brave-search"] env = { BRAVE_API_KEY = "BSAxxxxxxxxx" }

Auto-Review

Auto-Review automatically validates code changes before presenting them to you. It catches issues early so you can trust the output. It checks for:

  • Syntax errors and type issues
  • Test failures introduced by changes
  • Lint violations and formatting inconsistencies
  • Security vulnerabilities (basic static analysis)
  • Dependency conflicts and missing imports
  • Logical errors and edge case oversights

Subagents

Subagents allow Codex to spawn specialized child agents for complex tasks. Each subagent handles a focused subtask independently and reports back to the main agent, enabling parallel work on different parts of a problem.

Configure Amazon Bedrock

Amazon Bedrock allows you to route Codex's model inference through your AWS account. This is useful for enterprise compliance, cost management (usage counts toward AWS commitments), and data residency requirements.

Prerequisites

Before You Begin — Checklist

  • AWS Account — You need an active AWS account with billing enabled.
  • Bedrock Model Access — Request access to OpenAI models in the Bedrock Console under Model access. Enable openai.gpt-5.5 and/or openai.gpt-5.4.
  • IAM Permissions — Your AWS identity needs these permissions:
    • bedrock:InvokeModel
    • bedrock:InvokeModelWithResponseStream
  • AWS CLI (v2) — Install from docs.aws.amazon.com if you plan to use SDK credentials.
  • Codex Installed — Complete the installation steps from Section 3 above.

Model Access Approval

Bedrock model access requests can take a few minutes to a few hours to be approved. Submit your request before proceeding with the configuration steps below.

Step 1: Edit the Configuration File

Open (or create) the Codex configuration file at ~/.codex/config.toml and set the model provider to Amazon Bedrock:

~/.codex/config.toml
# ============================================ # OpenAI Codex — Model Provider Configuration # ============================================ # Set the model provider # Options: "openai" (default), "amazon-bedrock", "azure-openai" model_provider = "amazon-bedrock" # Specify which model to use # Must be enabled in your Bedrock console model = "openai.gpt-5.5"
Terminal — Create/Edit the Config File
# Create the .codex directory if it doesn't exist mkdir -p ~/.codex # Open the config file with your preferred editor nano ~/.codex/config.toml # or: code ~/.codex/config.toml # or: vim ~/.codex/config.toml

Bedrock Authentication

Codex supports multiple methods for authenticating with AWS. Choose the one that best fits your organization's security policies.

AWS SDK (Recommended)
Bedrock API Key
AWS SSO

Option A: AWS Configure (Access Keys)

The simplest method — configure standard AWS CLI credentials that Codex will automatically discover.

Terminal
# Run the AWS configure wizard aws configure # You'll be prompted for: AWS Access Key ID [None]: AKIAIOSFODNN7EXAMPLE AWS Secret Access Key [None]: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY Default region name [None]: us-east-1 Default output format [None]: json

Option B: Environment Variables

Set credentials directly as environment variables — useful for CI/CD or temporary sessions.

~/.bashrc or ~/.zshrc
# AWS Credentials via environment variables export AWS_ACCESS_KEY_ID="AKIAIOSFODNN7EXAMPLE" export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY" export AWS_DEFAULT_REGION="us-east-1" # Optional: Session token (for temporary/assumed-role credentials) export AWS_SESSION_TOKEN="FwoGZXIvYXdzE..."

Credential Priority

Codex follows the standard AWS SDK credential resolution order: environment variables → shared credentials file (~/.aws/credentials) → SSO/IAM Identity Center → EC2 instance profile.

If your organization provides a Bedrock API Key, you can use it directly without configuring the full AWS SDK.

~/.codex/.env
# Bedrock API Key authentication BEDROCK_API_KEY="br-xxxxxxxxxxxxxxxxxxxxxxxxxxxx" # AWS Region (required) AWS_REGION="us-east-1"

About Bedrock API Keys

Bedrock API Keys are generated in the AWS Bedrock Console under "API Keys." They provide a simpler authentication method but may have more restricted permissions compared to full IAM credentials. Best for individual developers or quick setup.

For organizations using AWS IAM Identity Center (formerly AWS SSO), configure an SSO profile that Codex will use.

Terminal — SSO Setup
# Configure a new SSO profile aws configure sso # Follow the interactive prompts: # SSO session name: my-org # SSO start URL: https://my-org.awsapps.com/start # SSO region: us-east-1 # SSO registration scopes: sso:account:access # CLI default client Region: us-east-1 # CLI profile name: my-bedrock-profile # Login via SSO (opens browser) aws sso login --profile my-bedrock-profile # Tell Codex which profile to use export AWS_PROFILE="my-bedrock-profile"
~/.aws/config — SSO Profile Example
[profile my-bedrock-profile] sso_session = my-org sso_account_id = 123456789012 sso_role_name = BedrockDeveloperAccess region = us-east-1 output = json [sso-session my-org] sso_start_url = https://my-org.awsapps.com/start sso_region = us-east-1 sso_registration_scopes = sso:account:access

Step 2: Create the .env File for Region

Create a .env file in the Codex config directory to specify your AWS region:

~/.codex/.env
# Required: AWS Region where Bedrock models are available AWS_REGION="us-east-1" # Optional: Custom endpoint URL (for VPC endpoints or proxies) # BEDROCK_ENDPOINT_URL="https://vpce-xxx.bedrock-runtime.us-east-1.vpce.amazonaws.com" # Optional: AWS Profile override (if not using default) # AWS_PROFILE="my-bedrock-profile"
Terminal — Quick One-Liner Setup
# Create the .env file with your region echo 'AWS_REGION="us-east-1"' > ~/.codex/.env # Verify it was created cat ~/.codex/.env

Feature Limitations on Bedrock

When using Codex through Amazon Bedrock, some features that rely on OpenAI-specific infrastructure are not available. Core coding functionality is fully supported.

Practical Impact

The unavailable features (Fast Mode, Web Search, Image Generation) are supplementary. All core coding functionality — including Goal Mode, Chat Mode, Subagents, MCP, and Auto-Review — works identically on Bedrock. Most developers won't notice a difference in their daily workflow.

Frequently Asked Questions

Got questions? Here are answers to the most common ones. Click any question to expand.

How do I switch back to OpenAI direct from Bedrock?

Simply update your ~/.codex/config.toml file:

# Switch back to OpenAI direct model_provider = "openai" model = "gpt-5.5"

Then restart Codex (quit and relaunch). You can also simply delete the model_provider line entirely — Codex defaults to OpenAI direct when no provider is specified. Use /status to confirm the change took effect.

Which AWS regions support OpenAI models on Bedrock?

As of June 2026, OpenAI models on Amazon Bedrock are available in:

  • us-east-1 (N. Virginia) — All models, full availability
  • us-west-2 (Oregon) — All models, full availability
  • eu-west-1 (Ireland) — All models, full availability
  • ap-northeast-1 (Tokyo) — GPT-5.5 only

Check the Bedrock model availability documentation for the latest updates. Set your chosen region in ~/.codex/.env.

How does pricing work on Bedrock vs OpenAI direct?

Pricing through Amazon Bedrock matches OpenAI's first-party pricing — you pay the same per-token rates. The key differences are billing and commitment benefits:

  • Billing: Charges appear on your AWS bill, not a separate OpenAI invoice
  • Commitments: Usage counts toward AWS Enterprise Discount Programs (EDP) and committed-use commitments
  • Credits: AWS promotional credits can be applied to Bedrock usage
  • Consolidated billing: Managed through AWS Organizations for multi-account setups
  • Cost allocation: Tag Bedrock usage for departmental chargebacks

This makes Bedrock especially attractive for enterprises with existing AWS spending commitments they need to fulfill.

Can I use Codex offline?

No. Codex requires an active internet connection. The AI models run in the cloud (either on OpenAI's infrastructure or AWS Bedrock), so connectivity is required for all code generation, analysis, and conversation features.

However, some local features work without connectivity:

  • Local file indexing and project context building can be cached
  • Skill definitions are stored and editable locally
  • Previously generated code persists on your filesystem
  • Session history is stored locally for review

If you lose connection mid-session, Codex will display an error and retry when connectivity returns. No work is lost.

What's the difference between Goal Mode and Chat Mode?

Goal Mode is for autonomous task execution. You describe what you want done, and Codex plans, implements, tests, and delivers the complete result without further input. Think of it as assigning a task to a capable junior developer.

Chat Mode is for interactive collaboration. You have a back-and-forth conversation, asking questions, discussing approaches, and iterating together. Think of it as pair programming.

AspectGoal Mode ⚡Chat Mode 💬
InteractionOne prompt → full deliveryMulti-turn conversation
Best forWell-defined, deliverable tasksExploration, learning, debugging
AutonomyHigh — minimal intervention neededLow — you guide each step
ExecutionRuns commands, creates/edits filesSuggests code for you to apply
OutputUnified diff ready to commitCode snippets and explanations

When to use which: If you can clearly state "I want X done" → Goal Mode. If you're thinking "I need help figuring out X" → Chat Mode.

How do I use MCP servers with Codex?

MCP (Model Context Protocol) servers extend Codex with external data sources and tools. Add them to your ~/.codex/config.toml:

# Database access — query schemas and data [[mcp_servers]] name = "postgres" command = "npx" args = ["@modelcontextprotocol/server-postgres", "postgresql://localhost:5432/mydb"] # Web search capability [[mcp_servers]] name = "brave-search" command = "npx" args = ["@modelcontextprotocol/server-brave-search"] env = { BRAVE_API_KEY = "BSAxxxxxxxxx" } # GitHub — PRs, issues, code search [[mcp_servers]] name = "github" command = "npx" args = ["@modelcontextprotocol/server-github"] env = { GITHUB_TOKEN = "ghp_xxxx" }

Popular MCP servers: PostgreSQL, MySQL, GitHub, Brave Search, Filesystem, Puppeteer (browser automation), Slack, Linear, and Sentry. Visit the MCP servers repository for a full list of community-maintained servers.

Codex vs Claude Code vs Cursor — which should I choose?

Here's a practical comparison for daily development:

Choose Codex if: You want maximum autonomy — describe a goal and let the AI handle everything. Best for well-defined tasks, multi-file changes, and teams already on AWS (Bedrock integration). The subagent system excels at complex, parallelizable work.

Choose Claude Code if: You need deep reasoning for hard problems — complex debugging, architectural decisions, thorough code reviews. Its long context window and reasoning depth are unmatched. CLI-only interface suits terminal-heavy workflows.

Choose Cursor if: You want AI seamlessly integrated into your editor. Real-time autocomplete, inline edits, and the familiar VS Code interface make it the lowest-friction option. Multi-model support (switch between providers) adds flexibility.

Many developers use 2-3 of these tools for different scenarios. They're complementary, not mutually exclusive.

How do I troubleshoot common errors?

Here are the most common issues and their solutions:

❌ "Model not found" or "Access denied"

  • Verify model access is enabled in the Bedrock console
  • Check model ID matches exactly: openai.gpt-5.5 (not gpt-5.5)
  • Ensure your region has that model available

❌ "Invalid credentials" or "ExpiredToken"

  • Run aws sts get-caller-identity to test credentials
  • If using SSO, run aws sso login to refresh the session
  • Check that AWS_REGION in your .env matches your credentials' region

❌ "Rate limit exceeded" / "ThrottlingException"

  • Request a quota increase in the Bedrock console (Service Quotas)
  • Add a fallback_model = "openai.gpt-5.4" in config.toml
  • Reduce concurrent subagents for Goal Mode tasks

❌ "Connection timeout" or "Network error"

  • Verify internet connectivity
  • Check if a VPN/proxy is blocking AWS endpoints
  • Try a different region to rule out regional issues
  • Ensure firewall allows HTTPS to bedrock-runtime.{region}.amazonaws.com

What are the security and privacy considerations?

Codex takes security seriously. Here's what you need to know:

🔐 Data Handling:

  • Your source code is sent to the model provider for inference but is not stored or used for model training
  • With Bedrock, data stays within your AWS account's data perimeter and VPC (if configured)
  • Session history is stored locally on your machine only
  • No code is persisted on OpenAI or AWS servers after the request completes

🔑 Authentication Security:

  • Credentials are stored in OS-native keychains (macOS Keychain, Windows Credential Manager, Linux Secret Service)
  • API keys in .env files should be protected with restricted permissions: chmod 600 ~/.codex/.env

🌐 Network Security:

  • All communication is encrypted via TLS 1.3
  • VPC endpoints are supported for Bedrock (keeps traffic on AWS backbone network)
  • No telemetry or usage data is collected without explicit opt-in

✅ Best Practices:

  • Add sensitive files to .codexignore (works like .gitignore) to prevent them from being sent to the model
  • Use temporary credentials or SSO instead of long-lived access keys
  • Always review code changes before committing — even from Goal Mode
  • Rotate API keys periodically
