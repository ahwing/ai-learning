核心概念深入

用费曼方法理解 RAG 的关键直觉

📖 为什么 RAG 有效？

类比：开卷考试 vs 闭卷考试

纯 LLM 回答问题就像闭卷考试——只能靠训练时记住的知识，记错了就会"幻觉"。RAG 则是开卷考试——允许模型翻阅参考资料再作答。你不需要把整本教科书背下来，只需要知道去哪里找、怎么找。RAG 的核心价值：让模型的回答有据可依，而不是凭记忆编造。

📐 Embedding 与向量相似度的直觉

类比：按主题把书放在书架上，相似的书自然靠在一起

Embedding 模型把文本映射到一个高维空间中的"坐标点"。语义相近的文本，坐标就靠得近。就像图书馆按主题分类——讲"权限配置"的段落和讲"安全策略"的段落会被放在相邻的位置，而讲"定价"的段落则在远处。搜索时，我们把问题也变成坐标，然后找最近的邻居——这就是向量检索的本质。

💡 余弦相似度衡量两个向量方向的一致性（-1 到 1）。方向越一致 → 语义越相似。不关心向量长度，只关心"朝哪个方向"。
🌍 现实案例：Amazon Bedrock Knowledge Base、LangChain、LlamaIndex 是最流行的 RAG 框架。企业用它们让 AI 理解内部文档——比如 AWS 文档助手可以回答特定于你账户配置的问题。

✂️ 分块策略的权衡

Chunk 太大和太小都有问题，需要找到平衡点：

❌ Chunk 太大
🌍 现实案例：OpenAI 的 text-embedding-3-large 和 Cohere Embed v3 是目前最强的嵌入模型。向量数据库方面，Pinecone、Weaviate、Amazon OpenSearch 是主流选择。
  • 检索不精确，混入无关内容
  • 占用 LLM 上下文窗口
  • 向量表示被"稀释"
❌ Chunk 太小
  • 丢失上下文，断章取义
  • 需要更多 chunk 才能覆盖答案
  • 增加检索和存储成本
✅ 实践建议
  • 通用场景：300-500 tokens
  • 重叠 50-100 tokens 保持连贯
  • 按语义边界（段落/章节）切分
  • 保留元数据（标题、来源）

🏆 为什么需要重排序？

两阶段策略：快速粗筛 → 精确打分

向量搜索很快（毫秒级），但它是"双塔模型"——问题和文档分别编码，无法看到它们之间的细粒度交互。Reranker 是"交叉编码器"——把问题和文档拼在一起输入，能捕捉词级别的匹配关系，精度远高于纯向量距离。但它慢（每对都要过一次模型），所以只对粗排的 Top-50 做精排。这就像招聘：先用简历关键词筛出 50 人（快），再逐个面试选出 5 人（慢但准）。

替代方案与优化

超越基础 RAG 的进阶模式

🚀 高级 RAG 模式

HyDE

先让 LLM 生成一个"假设性答案"，再用这个答案去检索。假设答案比问题更接近真实文档的表述方式。

Multi-Query

将用户问题改写为多个不同角度的查询，分别检索后合并去重。覆盖更全面，减少遗漏。

Contextual Compression

检索后用 LLM 压缩每个 chunk，只保留与问题相关的部分。减少噪音，节省上下文窗口。

🔀 混合搜索 (Hybrid Search)

向量搜索擅长语义匹配（"怎么配置权限" ≈ "access control setup"），但对精确关键词（错误码、产品名）不敏感。BM25 关键词搜索擅长精确匹配但不懂同义词。混合搜索结合两者优势：用 RRF (Reciprocal Rank Fusion) 或加权分数合并两路结果，兼顾语义理解和精确匹配。

🕸️ Graph RAG

在向量检索基础上引入知识图谱。先从文档中抽取实体和关系构建图，检索时不仅找相似 chunk，还沿着图的边找到关联实体。特别适合需要多跳推理的问题（"A 公司的 CEO 毕业于哪所大学？"需要先找到 CEO 是谁，再找其教育背景）。

🤖 Agentic RAG

让 Agent 自主决定何时检索、检索什么、是否需要再次检索。不是每个问题都需要 RAG（"1+1=?" 不需要），Agent 可以判断：直接回答 / 检索一次 / 检索后发现不够再换个 query 检索 / 调用其他工具。这是 RAG 从"管线"进化为"智能体"的方向。

⚖️ Fine-tuning vs RAG：何时用哪个？

选 RAG 当：
  • 知识频繁更新
  • 需要引用来源
  • 数据量大且多样
  • 需要精确事实性回答
  • 快速部署，无需训练
选 Fine-tuning 当：
  • 需要特定风格/格式输出
  • 领域术语和推理模式固定
  • 延迟要求极高（省去检索）
  • 知识相对稳定不常变
  • 需要模型"内化"某种能力

💡 两者不互斥——可以 Fine-tune 一个更擅长利用 context 的模型，再配合 RAG 使用。

常见问题

实践中最常遇到的 RAG 问题

Q: 文档频繁更新怎么办？

增量索引：不要每次全量重建。为每个 chunk 记录来源文档的版本/时间戳，文档更新时只重新处理变化的部分。可以用 hash 检测内容是否真的变了。
TTL 策略：为 chunk 设置过期时间，定期清理过时内容。
版本标记：保留多版本，检索时优先返回最新版本，但允许用户查询历史版本。

Q: Chunk size 应该设多大？

没有万能答案，取决于文档类型和使用场景：
• 技术文档/FAQ：200-400 tokens（信息密度高，小块更精确）
• 叙事性文档/报告：400-800 tokens（需要更多上下文）
• 代码：按函数/类切分，而非固定长度
• 通用建议：从 512 tokens + 50 overlap 开始，用评估指标调优。Parent-child 策略（小块检索，返回大块上下文）是个好折中。

Q: RAG vs Fine-tuning vs 长上下文：怎么选？

• 长上下文（128K+ tokens）：文档少且固定时最简单，直接塞进 prompt。但成本高、延迟大、"大海捞针"效果下降。
• RAG：文档多/常更新/需要引用来源时首选。额外复杂度换来可扩展性和可解释性。
• Fine-tuning：需要改变模型行为/风格/推理方式时用。不适合注入大量事实知识（容易遗忘）。
• 组合使用：Fine-tune 让模型更擅长利用 context + RAG 提供 context + 长上下文作为兜底。

Q: 如何评估 RAG 质量？

分两层评估：
检索质量：Recall@K（正确 chunk 是否被召回）、MRR（正确答案排第几）、NDCG。
生成质量：
• Faithfulness（忠实度）：回答是否基于检索到的内容，有没有编造？
• Relevance（相关性）：回答是否切题？
• Completeness（完整性）：是否覆盖了所有要点？
工具推荐：RAGAS、TruLens、LangSmith、Arize Phoenix。用 LLM-as-Judge 自动评估 + 人工抽检。

Q: 如何处理多模态文档（表格、图片）？

• 表格：将表格转为结构化文本（Markdown 表格或自然语言描述），或用专门的 Table QA 模型。保留表头信息作为 chunk 的一部分。
• 图片/图表：用多模态模型（GPT-4V、Claude Vision）生成图片描述，将描述文本作为 chunk 索引。或使用 CLIP 等多模态 embedding 直接索引图片向量。
• PDF 布局：用 Layout-aware 解析器（如 Unstructured、DocTR）保留文档结构，区分标题、正文、表格、图注。
• 混合检索：文本 chunk 和图片 chunk 分别索引，检索时合并排序。

🚀 RAG 优化技巧（AWS 视角）

🔀 Hybrid Search（混合搜索）

结合关键词搜索（BM25）与语义向量搜索。OpenSearch 原生支持 hybrid query，用 RRF 或加权分数融合两路结果。
→ 精确术语（错误码、产品名）不被遗漏

🏆 Reranking（重排序）

Bedrock 已集成 Cohere Rerank 模型。粗排 Top-50 → 精排 Top-5，显著提升检索准确率。
→ 在 Knowledge Base API 中一行配置即可启用

🏷️ Metadata Filtering（元数据过滤）

为 chunk 附加元数据标签（部门、日期、文档类型），检索时按条件过滤，缩小搜索范围。
→ 支持 S3 元数据文件自动关联

👪 Parent-Child Chunking（层次分块）

小 chunk 用于精确检索（命中率高），返回时包含其 parent chunk 的完整上下文。兼顾精度和上下文完整性。
→ Bedrock 支持 Hierarchical chunking 策略

🧩 Query Decomposition（查询分解）

复杂问题拆分为多个子查询分别检索再合并。配合 Bedrock Agent 的多步推理能力实现。
→ "对比 A 和 B 的优缺点" → 分别检索 A、B 再综合

💡 实践建议：从 Bedrock Knowledge Base 默认配置开始 → 评估检索质量 → 逐步启用 Hybrid Search + Reranking → 按需添加 Metadata Filtering 和自定义 Chunking。不要一上来就做全部优化，先跑通再迭代。
