深入理解 Guardrails 机制

用直觉理解每种防护能力的原理

🎯 内容过滤的分级机制

类比：机场安检分级 — 不是所有东西都禁止，而是按风险分级处理

Bedrock Guardrails 的内容过滤不是简单的"有/无"判断，而是把每个维度（仇恨、暴力、性、侮辱、不当行为）的严重程度分为 NONE、LOW、MEDIUM、HIGH 四档。你可以为每个维度独立设置阈值 — 比如医疗场景可以容忍"性"维度的 MEDIUM（讨论生殖健康），但"暴力"维度设为零容忍。

💡 关键区分：输入过滤和输出过滤可以设不同阈值。输入侧宽松一些（用户可能在描述问题），输出侧严格一些（模型不应生成有害内容）。

🔐 PII 检测与脱敏策略

两种处理模式：BLOCK（拦截）vs ANONYMIZE（脱敏后放行）

🚫 BLOCK 模式

检测到 PII 即拒绝整个请求。适用于绝对不能接触敏感数据的场景（如公开的客服机器人）。

🔄 ANONYMIZE 模式

将 PII 替换为占位符（如 {NAME_1}、{EMAIL_1}）后继续处理，输出时反向替换。模型永远看不到真实数据。

📐 上下文事实核查 (Contextual Grounding)

类比：论文查重 — 每句输出都要能在参考文献中找到依据

这是 RAG 场景中最重要的 Guardrail。它检查模型输出中的每个声明（claim）是否能在提供的参考文档（grounding source）中找到支持证据。如果某个声明的相关性得分低于阈值（默认 0.7），该声明会被标记或移除。这直接对抗了 LLM 的"幻觉"问题 — 模型不能凭空编造源文档中不存在的信息。

💡 grounding_threshold 设置在 0~1 之间。0.7 是推荐值；对于金融/医疗等高风险场景建议 ≥ 0.85。

🧮 Automated Reasoning — 最新的形式化验证能力

类比：数学证明检验 — 不仅检查答案，还验证推理过程

Automated Reasoning 超越了"事实核查"，它将业务规则转化为形式化逻辑，然后用自动定理证明器验证 LLM 的每一步推理是否符合规则。例如：保险理赔场景，规则是"免赔额 = max(500, 保额×10%)"，如果 LLM 计算出错误的免赔额，Automated Reasoning 会用数学方法证明输出错误并拒绝。这是唯一能提供数学级别确定性保证的 Guardrail。

🏗️ 生产环境最佳实践

多层防御 (Defense in Depth) 策略

🌐 第一层：网络边界
  • CloudFront + WAF 过滤恶意请求
  • 速率限制防止滥用
  • IP 白名单/地理围栏
  • 请求体大小限制
🛡️ 第二层：Guardrails 输入检查
  • Prompt Injection 检测
  • PII 脱敏 (ANONYMIZE)
  • 禁止话题拦截
  • 关键词/正则黑名单
🤖 第三层：模型层安全
  • System Prompt 注入安全指令
  • 温度调低减少随机性
  • 使用 RLHF 对齐的模型
  • 限制 max_tokens 防过长输出
✅ 第四层：Guardrails 输出检查
  • 有害内容过滤（更严格阈值）
  • Contextual Grounding 防幻觉
  • Automated Reasoning 验证
  • PII 泄露最终拦截
💡 核心原则：永远不要只依赖单一防线。即使 Guardrails 完美工作，仍然需要 WAF、认证、日志审计、人工审核等多层保护。安全是一个系统性问题，不是一个 API 调用能解决的。
