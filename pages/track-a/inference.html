KV Cache：推理加速的核心

避免重复计算，将 O(n²) 降为 O(n)

❌ 无 KV Cache
🌍 现实案例：vLLM 的 PagedAttention 技术让推理吞吐量提升了 2-24x，已成为生产环境部署 LLM 的事实标准。HuggingFace TGI 和 NVIDIA TensorRT-LLM 也采用了类似思路。
生成 token 5 时：
重新计算 token 1,2,3,4 的 K,V
生成 token 6 时：
重新计算 token 1,2,3,4,5 的 K,V
→ 大量重复计算，O(n²) 复杂度
✓ 标准做法
✅ 有 KV Cache
生成 token 5 时：
从 cache 读取 token 1-4 的 K,V ⚡
只计算 token 5 的 Q,K,V
→ 每步只算 1 个 token，O(n) 复杂度
代价：显存占用随序列长度线性增长

📊 推理性能关键指标
• TTFT (Time to First Token) — 首 token 延迟，用户感知的响应速度
• TPS (Tokens per Second) — 生成速度，decode 阶段的吞吐
• Throughput — 系统总吞吐量（tokens/s across all requests）
• 显存占用 — 模型权重 + KV Cache + 激活值
• Cost/Token — 每 token 推理成本，决定商业可行性

🧠 核心概念深入

用费曼式类比，直觉理解推理的本质

🔗 为什么自回归生成是顺序的？
💡 类比：写一句话
想象你在写"今天天气真好"。你必须先写"今"，才能决定下一个字是"天"而不是"年"。每个字都依赖前面所有字的语境。
• 每个 token 的概率分布取决于所有前文
• P(x₊ | x₁, x₂, ..., x₊₋₁) — 条件概率链
• 这就是为什么生成不能并行：下一个词还没"出生"
• 训练时可以并行（Teacher Forcing），推理时不行
⚡ Prefill vs Decode 的本质区别
💡 类比：读书 vs 写作
Prefill 像快速阅读一整页（并行处理已知文本），Decode 像一个字一个字地创作（每步依赖上一步）。读书快因为内容已知，写作慢因为要"想"。
• Prefill：Compute-bound，GPU 算力是瓶颈
• Decode：Memory-bound，显存带宽是瓶颈
• Prefill 处理 N 个 token 只需 1 次前向传播
• Decode 生成 N 个 token 需要 N 次前向传播
• 优化方向完全不同：一个要更多算力，一个要更快的内存
🌡️ Temperature 与采样的直觉
💡 类比：创造力旋钮
Temperature 就像一个"创造力旋钮"。调低（0.1）= 保守谨慎，总选最可能的词；调高（1.5）= 天马行空，可能出奇迹也可能说胡话。
• T→0：概率分布变尖锐，趋近 greedy（确定性）
• T=1：原始分布，模型"本色出演"
• T>1：分布变平坦，低概率词也有机会
• 数学本质：logits / T → softmax，T 越大越均匀
• 实践：代码生成用 0.2，创意写作用 0.8-1.0
🗜️ 为什么量化几乎不损失质量？
💡 类比：JPEG 压缩
就像 JPEG 把图片从 10MB 压到 500KB，人眼看不出区别。模型权重中大量信息是"冗余"的，用更少的比特表示，模型的"理解力"几乎不变。
• 神经网络对噪声天然鲁棒（训练时就有 dropout）
• 权重分布集中，离群值少 → 量化误差小
• FP16→INT4：每个权重从 16bit 变 4bit，省 4× 显存
• 关键技巧：按 group 量化 + 保留离群值精度
• 实测：INT4 在大多数 benchmark 上损失 < 1%

🚀 替代方案与优化

超越标准自回归：前沿加速与生成技术

🔮 投机解码深入解析
Draft-Verify 范式：
1. 小模型（Draft）快速生成 γ 个候选 token
2. 大模型（Target）一次前向传播并行验证所有候选
3. 从左到右接受匹配的 token，在第一个不匹配处重新采样
为什么是无损的？
通过修正后的拒绝采样（modified rejection sampling），数学上保证输出分布与大模型独立生成完全一致。接受的 token 概率 = min(1, p_target/p_draft)。
变体：Medusa（多头并行猜测）、EAGLE（特征级草稿）、Lookahead Decoding（Jacobi 迭代）
⚡ 非自回归生成尝试
打破顺序生成的枷锁：
• 并行解码：一次预测多个位置，迭代修正
• Diffusion LM：将文本生成建模为去噪过程
• Mask-Predict：先生成全部（含 [MASK]），再迭代填充
• CMLM：条件 Masked LM，多轮并行解码
现状与挑战
质量仍不及自回归模型，尤其在长文本连贯性上。但在特定场景（翻译、语音合成）已有成功应用。文本扩散模型（如 MDLM）是活跃研究方向。
💾 KV Cache 压缩技术
减少 KV Cache 显存占用：
• GQA（Grouped Query Attention）：多个 Q 头共享 KV 头
• MQA（Multi-Query Attention）：所有 Q 头共享 1 组 KV
• MLA（Multi-head Latent Attention）：低秩压缩 KV
• Sliding Window：只保留最近 W 个 token 的 KV
• H2O（Heavy Hitter Oracle）：只保留高注意力分数的 KV
效果对比
GQA（Llama 2 用）：KV Cache 减少 4-8×
MLA（DeepSeek-V2 用）：压缩比高达 93%
Sliding Window（Mistral 用）：固定显存，支持无限长度
📐 结构化生成
让 LLM 输出严格符合格式：
• 约束解码：在采样时屏蔽不合法 token
• Grammar-guided：用 CFG/正则约束输出
• JSON Mode：保证输出是合法 JSON
• Outlines/Guidance：开源结构化生成框架
工作原理
在每一步解码时，根据当前状态（已生成的部分）计算哪些 token 是"合法的下一步"，将不合法 token 的 logit 设为 -∞。零额外延迟，100% 格式保证。
应用：API 调用、工具使用、数据提取、Agent 决策

❓ 常见问题

关于 LLM 推理的高频疑问

Q1：为什么 LLM 推理这么贵？
成本构成：
• GPU 硬件：A100/H100 单卡 $1-3/小时，70B 模型需要多卡
• 显存占用：模型权重 + KV Cache + 激活值，大模型需要大量 HBM
• 低利用率：Decode 阶段 GPU 算力利用率仅 5-15%（memory-bound）
• 序列长度：KV Cache 随 context 线性增长，长对话成本翻倍
💡 降本路径：量化（4× 省显存）+ Batching（10× 提吞吐）+ 投机解码（2-3× 加速）
Q2：如何选择延迟优化 vs 吞吐量优化？
• 优化延迟（单请求快）：实时对话、代码补全、交互式应用
   → 投机解码、Tensor Parallel、更快的硬件
• 优化吞吐（总处理量大）：批量处理、离线分析、API 服务
   → Continuous Batching、大 batch size、Pipeline Parallel
• 两者兼顾：Prefill-Decode 分离（PD Disaggregation）
   → 不同机器分别优化 Prefill 和 Decode
💡 经验法则：面向用户 → 优先延迟；面向系统 → 优先吞吐
Q3：什么时候用量化 vs 蒸馏？
• 量化：快速部署，不需要重新训练，精度损失小
   → 适合：已有大模型想快速上线，资源有限
   → 局限：INT4 以下质量下降明显，不改变计算量
• 蒸馏：训练小模型模仿大模型，从根本上减少参数
   → 适合：有训练资源，追求极致性价比
   → 局限：需要训练数据和算力，效果有上限
💡 最佳实践：先蒸馏得到小模型，再量化部署 = 双重压缩
Q4：为什么上下文越长，生成越慢？
• KV Cache 增大：每个 token 都要 attend 到所有历史 token
• Attention 计算：标准 attention O(n²)，即使有 FlashAttention 也是 O(n)
• 显存带宽：更大的 KV Cache 需要更多时间从 HBM 读取
• Prefill 变慢：长 prompt 的 prefill 时间线性增长
💡 缓解方案：Sliding Window Attention、稀疏注意力、KV Cache 量化、Ring Attention（分布式长序列）
Q5：推理优化的未来方向？
• 硬件专用化：Groq LPU、Cerebras WSE — 专为推理设计的芯片
• 模型架构革新：Mamba/SSM 线性复杂度、混合架构
• 动态计算：Early Exit、MoE 稀疏激活 — 简单问题用更少计算
• 编译优化：torch.compile、Triton 自定义 kernel — 软件层极致优化
• 缓存与复用：Semantic Cache、Prompt Cache — 避免重复计算
💡 趋势：推理成本每年下降 10×，最终让大模型像水电一样便宜
