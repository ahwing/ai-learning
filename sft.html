监督微调 (SFT)

从基座模型到指令遵循 — 让模型学会"听话"

🌍 现实案例：InstructGPT (OpenAI, 2022) 用仅约 13K 条人工标注的指令数据，就让 1.3B 参数的模型超越了 175B 的 GPT-3。这证明了「少量高质量数据 > 海量低质量数据」。

SFT 数据格式详解

不同格式的指令数据，决定了模型学到的能力

单轮对话格式
<|system|> You are a helpful assistant.
<|user|> 解释什么是梯度下降
<|assistant|> 梯度下降是一种优化算法...
Loss 只计算 assistant 部分（masked loss）
🌍 现实案例：ChatGPT、Claude、Gemini 上线前都经过了精心设计的 SFT 阶段。SFT 数据的质量直接决定了模型的"人格"——是否有礼貌、是否会拒绝有害请求、回答风格如何。
多轮对话格式
<|user|> 写一个 Python 排序函数
<|assistant|> def sort_list(arr):...
<|user|> 加上类型注解
<|assistant|> def sort_list(arr: list[int])...
多轮上下文让模型学会追问和修改

🌍 现实案例：Alpaca (Stanford) 用 GPT-4 生成了 52K 条指令数据，成本仅 $500。这开创了"用强模型训练弱模型"的 Self-Instruct 范式，让开源社区快速跟进。

💡 SFT 关键洞察
1. 质量 >> 数量 — LIMA 论文证明 1000 条高质量数据就能产生很好的效果
2. 多样性很重要 — 覆盖不同任务类型（问答、写作、代码、数学、角色扮演）
3. 格式一致性 — 统一的 chat template 让模型学会结构化输出
4. Masked Loss — 只在 assistant 回复上计算 loss，不学 user 的提问方式
5. 过拟合风险 — 数据少 + epoch 多容易过拟合，通常只训 1-3 epochs

🧠 核心概念深入

用费曼式直觉理解 SFT 的关键机制

🎯 为什么 Masked Loss 有效？
直觉类比：想象你是老师在批改考试卷。你只给学生的答案打分，而不会给题目本身打分——因为题目是你出的，不需要学生"学会出题"。
📝 题目（user prompt）→ mask=0，不计算 loss，模型不需要学会"怎么提问"
✍️ 答案（assistant response）→ mask=1，计算 loss，模型专注学习"怎么回答"
🎓 效果→ 训练效率更高，模型不会学到"模仿用户说话"的坏习惯
📎 LoRA 简单解释
直觉类比：想象你有一本百科全书（预训练模型的权重）。与其重写整本书，不如在关键页面贴上便利贴（低秩矩阵），写上修正和补充。读的时候，原文 + 便利贴 = 更好的内容。
原始权重 W（冻结，不动）+ ΔW = A × B（只训练这个小矩阵）
关键洞察：微调时权重的变化量 ΔW 是低秩的（信息量不大），所以可以用两个小矩阵 A·B 来近似
实际效果：70B 模型只需训练 ~0.1% 的参数，显存从 140GB 降到 16GB，效果损失极小
💥 灾难性遗忘：为什么会发生？如何预防？
直觉类比：就像一个钢琴家为了学吉他，练习太猛导致忘了怎么弹钢琴。神经网络的参数是共享的——新任务的梯度更新会覆盖旧知识的编码。
❌ 导致遗忘的因素
• 学习率过高
• 训练 epoch 过多
• 数据分布太窄
• 全参数微调
✅ 预防策略
• 使用 LoRA（冻结主干）
• 低学习率 (1e-5~2e-5)
• 混入预训练数据 (5-10%)
• 早停 (early stopping)
⚖️ SFT 的"对齐税"——可能失去什么？
SFT 让模型变得"听话"，但这不是免费的。模型在获得指令遵循能力的同时，可能会牺牲一些原始能力：
可能退化的能力
• 开放式创意写作（变得模板化）
• 少数语言的流利度
• 某些推理任务的灵活性
• 长文本续写的连贯性
缓解方法
• 数据多样性（覆盖各种任务）
• 混入通用语料继续训练
• 控制训练步数，不过度拟合
• 使用 LoRA 保留原始能力
💡 这就是为什么 SFT 后还需要 RLHF/DPO——在保持能力的同时进一步对齐

🧬 数据生成策略
Self-Instruct
让模型自己生成指令和回答，再过滤低质量样本。成本低但质量参差不齐。代表：Alpaca
Evol-Instruct
渐进式增加指令复杂度（加约束、深化、拓宽），生成更有挑战性的训练数据。代表：WizardLM
RLHF-as-SFT
用 RLHF 模型的输出作为 SFT 数据（蒸馏对齐行为）。跳过 RL 训练的不稳定性。代表：Zephyr
Rejection Sampling
对同一指令生成多个回答，用奖励模型选最好的作为训练数据。质量高但计算成本大。
🎯 单任务 SFT
• 只针对一个特定任务（如代码生成、医疗问答）
• 数据量需求少（1K-10K 即可）
• 该任务效果极好，但通用性差
• 适合：垂直领域应用、特定业务场景
🌐 多任务 SFT
• 混合多种任务类型（问答+代码+写作+数学）
• 数据量需求大（50K-500K）
• 通用能力强，各任务互相促进
• 适合：通用助手、ChatGPT 类产品
🛠️ 指令微调实用技巧
数据去重
语义级去重（不只是字面重复），避免模型对某些模式过拟合。使用 embedding 相似度过滤。
回复长度控制
训练数据中回复长度要多样化。全是长回复→模型啰嗦；全是短回复→模型过于简略。
Rejection Sampling
生成 N 个候选回复，用奖励模型/规则筛选最佳。相当于"只用最好的答案训练"。
数据配比调优
不同任务类型按比例混合。代码/数学数据占比过高会影响对话能力，需要反复实验找到平衡点。

❓ 常见问题

SFT 实践中最常遇到的疑问

Q1: SFT 需要多少数据才够？
简短回答：取决于目标。通用助手需要 50K-500K 条多样化数据；垂直领域 1K-10K 条高质量数据就能见效。
关键原则：LIMA 论文表明，1000 条精心策划的数据就能让模型学会对话格式。但要覆盖更多能力（代码、数学、多语言），数据量需要相应增加。质量永远比数量重要——100 条垃圾数据不如 10 条精品数据。
Q2: SFT 会让模型变差吗？
会的，主要两种情况：
① 过拟合 — 数据太少或 epoch 太多，模型开始"背答案"而非学会泛化。表现：训练集上完美，新问题上胡说。
② 灾难性遗忘 — 学习率太高或数据分布太窄，模型忘记预训练学到的知识。表现：MMLU 等基准分数下降。
预防：监控验证集 loss、定期跑基准测试、使用 LoRA、控制训练步数。
Q3: LoRA Rank 怎么选？
经验法则：
• rank=8-16 — 简单任务（格式学习、风格迁移），参数少，训练快
• rank=32-64 — 主流选择，适合大多数 SFT 场景，平衡效果和效率
• rank=128-256 — 复杂任务（多语言、代码+对话混合），接近全参数效果
直觉：rank 越高，"便利贴"能记录的信息越多，但训练成本也越高。如果 rank=16 效果已经饱和，加到 64 也不会更好。建议从 32 开始，根据验证集效果调整。
Q4: 什么时候该用全参数微调 vs LoRA？
选全参数微调：① 有充足 GPU 资源（8×A100 以上）；② 需要模型深度改变行为（如从英文模型训成中文模型）；③ 追求最后 1-2% 的效果提升。
选 LoRA：① 资源有限（单卡或少量卡）；② 任务相对简单（学格式、学风格）；③ 需要快速迭代实验；④ 想保留模型原始能力。
实际情况：90% 的场景 LoRA 就够了。全参数微调主要用于训练基础 chat 模型（如 LLaMA → LLaMA-Chat）这种大规模行为改变。
Q5: 除了跑 benchmark，怎么评估 SFT 质量？
多维度评估策略：
① 人工盲评 — 让标注员在不知道哪个是新模型的情况下对比打分，最可靠但成本高
② LLM-as-Judge — 用 GPT-4/Claude 对模型输出打分（MT-Bench 方式），成本低、相关性高
③ 对话质量抽检 — 随机抽 100-200 条真实用户问题，人工检查回答质量、安全性、格式
④ 回归测试 — 维护一组"必须答对"的核心测试用例，确保新版本不退化
⑤ 用户反馈 — 上线后收集 👍👎 反馈，持续监控满意度变化
